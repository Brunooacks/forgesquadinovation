# ForgeSquad Production Output Standard

> **SHARED FILE** — applies to ALL IDEs. Do not add IDE-specific logic here.

This document defines the mandatory output standard for projects generated by ForgeSquad squads.
Every project output must be deployable to production with maturity, security, and observability.

## Standard Project Structure

All projects generated by ForgeSquad squads MUST follow this directory structure:

```
project-root/
├── src/                        # Source code
│   ├── main/                   # Application code
│   └── config/                 # Configuration files
├── tests/                      # Test suites
│   ├── unit/                   # Unit tests (80% coverage target)
│   ├── integration/            # Integration tests (60% coverage target)
│   └── e2e/                    # End-to-end tests (critical paths)
├── infra/                      # Infrastructure as Code
│   ├── terraform/              # or bicep/, cloudformation/
│   └── k8s/                    # Kubernetes manifests (if applicable)
├── .github/
│   └── workflows/              # CI/CD pipelines
│       ├── ci.yml              # Build, test, lint, security scan
│       └── cd.yml              # Deploy to staging/production
├── docker/
│   ├── Dockerfile              # Development Dockerfile
│   ├── Dockerfile.prod         # Production multi-stage Dockerfile
│   └── docker-compose.yml      # Local development stack
├── docs/
│   ├── architecture.md         # Architecture overview + C4 diagrams
│   ├── adrs/                   # Architecture Decision Records
│   ├── runbook.md              # Operational runbook
│   └── api.md                  # API documentation (OpenAPI reference)
├── scripts/
│   ├── setup.sh                # Development environment setup
│   ├── migrate.sh              # Database migration runner
│   └── seed.sh                 # Test data seeding
├── .env.example                # Environment variables template (NO secrets)
├── Makefile                    # Standardized commands
├── README.md                   # Getting started + architecture summary
├── CONTRIBUTING.md             # Contribution guidelines
└── .gitignore                  # Language/framework-specific ignores
```

## Production-Ready Checklist

This checklist is validated by the QA Engineer and Reviewer agents before deployment.
Items marked with ** are VETO conditions — failing these blocks deployment.

### Infrastructure & Deployment
- [ ] ** CI/CD pipeline configured (build → test → lint → security scan → deploy)
- [ ] ** Dockerfile.prod uses multi-stage build (builder → runtime)
- [ ] ** Non-root user in production container
- [ ] Docker Compose for local development with all dependencies
- [ ] Infrastructure as Code covers all cloud resources
- [ ] Environment-specific configs (dev, staging, prod) separated

### Application Health
- [ ] ** Health check endpoint: `GET /health` (returns 200 when healthy)
- [ ] ** Readiness endpoint: `GET /ready` (returns 200 when ready to serve traffic)
- [ ] ** Graceful shutdown implemented (drain connections, finish in-flight requests)
- [ ] Startup probe configured (for slow-starting applications)

### Observability
- [ ] ** Structured logging (JSON format, correlation IDs)
- [ ] ** Request/response logging with timing
- [ ] Metrics endpoint (Prometheus format or cloud-native)
- [ ] Distributed tracing configured (OpenTelemetry)
- [ ] Dashboard defined (Grafana/CloudWatch/Datadog)
- [ ] Alerting rules defined (error rate, latency, saturation)

### Security
- [ ] ** No hardcoded secrets in code or config files
- [ ] ** Secrets managed via vault (AWS SM, Azure KV, HashiCorp Vault)
- [ ] ** SAST scan passes with zero critical/high findings
- [ ] ** Dependency scan passes with zero critical findings
- [ ] Input validation on all API endpoints
- [ ] Authentication and authorization implemented
- [ ] CORS policy configured
- [ ] Rate limiting on public endpoints
- [ ] Security headers configured (HSTS, CSP, X-Frame-Options)

### Data
- [ ] ** Database migrations versioned and reversible (up + down)
- [ ] ** .env.example documents all required environment variables
- [ ] Connection pooling configured
- [ ] Database indices for common queries
- [ ] Backup strategy documented

### Quality
- [ ] ** Unit test coverage >= 80%
- [ ] ** Integration test coverage >= 60%
- [ ] ** All tests pass in CI
- [ ] ** Linting passes with zero errors
- [ ] ** Type checking passes (if applicable)
- [ ] E2E tests cover critical user paths
- [ ] Performance tests baseline established

### Documentation
- [ ] ** README with setup instructions (clone → run in < 5 minutes)
- [ ] ** API documented (OpenAPI/Swagger for REST, schema for GraphQL)
- [ ] ** Runbook with operational procedures (deploy, rollback, incident response)
- [ ] Architecture documented with diagrams
- [ ] ADRs for all significant decisions

### Resilience
- [ ] Circuit breaker on external service calls
- [ ] Retry with exponential backoff for transient failures
- [ ] Timeout configured for all external calls
- [ ] Fallback behavior defined for dependency failures

## Makefile Standard

Every project MUST have a Makefile with these targets:

```makefile
.PHONY: help build test lint format migrate seed docker-build docker-run deploy-staging deploy-prod

help:           ## Show available commands
build:          ## Build the application
test:           ## Run all tests
test-unit:      ## Run unit tests only
test-integration: ## Run integration tests only
lint:           ## Run linter
format:         ## Format code
migrate:        ## Run database migrations
seed:           ## Seed test data
docker-build:   ## Build Docker image
docker-run:     ## Run with Docker Compose
deploy-staging: ## Deploy to staging
deploy-prod:    ## Deploy to production (requires confirmation)
```

## Enforcement

### During Pipeline Execution
- The **QA Engineer** agent validates the checklist after implementation steps
- The **Tech Lead** agent validates during code review steps
- The **Architect** agent validates during architecture review steps

### At Deploy Step
- The deploy step (`step-18-deploy.md` or equivalent) MUST run the production-ready
  checklist validation before allowing deployment
- Veto conditions (marked with **) trigger automatic rejection
- Non-veto items generate warnings but don't block deployment

### Scoring
```
Production Readiness Score:
- Infrastructure (25%): CI/CD, Docker, IaC, config separation
- Security (25%): Secrets, scans, auth, headers
- Quality (25%): Tests, lint, types, coverage
- Observability (15%): Logging, metrics, tracing, alerts
- Documentation (10%): README, API docs, runbook, ADRs

Score >= 90: Production Ready ✅
Score 75-89: Ready with Warnings ⚠️
Score < 75: Not Ready ❌ (veto deployment)
```

## Templates

Production scaffolding templates are available in `_forgesquad/core/templates/`:

- `dockerfile.prod.template` — Multi-stage production Dockerfile
- `github-actions-ci.template` — CI pipeline (build, test, lint, scan, deploy)
- `makefile.template` — Standardized Makefile with all targets
- `docker-compose.template` — Local dev stack (app, db, redis, monitoring)
- `env-example.template` — .env.example with categorized variables
- `runbook.template` — Operational runbook template

These templates are applied by the Architect during squad creation (Phase 5)
and refined by the Dev Backend agent during implementation steps.
